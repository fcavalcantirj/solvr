# Phase 8: Referral Tracking + Join Flow — Execution Plan

**Wave:** 08-01 (single wave)
**Requirements:** REF-03, PAGE-01
**Depends on:** Phase 6 complete (referrals table, referral_code column exist)

---

## Objective

Wire referral attribution into the registration flow. When a user signs up via `/join?ref=CODE`, the backend records who referred them in the `referrals` table. Invalid/unknown codes are silently ignored — registration always succeeds.

---

## Tasks

### Task 1: Add `CreateReferral` to ReferralRepository

**File:** `backend/internal/db/referral.go`

Add method:
```go
func (r *ReferralRepository) CreateReferral(ctx context.Context, referrerID, referredID string) error
```

- INSERT INTO `referrals` (id, referrer_id, referred_id, created_at) VALUES (gen_random_uuid(), $1, $2, NOW())
- Return error on failure (caller decides whether to log-and-continue or propagate)

Also add a lookup helper:
```go
func (r *ReferralRepository) FindUserIDByReferralCode(ctx context.Context, code string) (string, error)
```

- SELECT id FROM users WHERE referral_code = $1 AND deleted_at IS NULL
- Returns `db.ErrNotFound` if code doesn't match any user

**Tests (TDD):** `backend/internal/db/referral_test.go`
- Test CreateReferral inserts a row and CountByReferrer returns 1
- Test CreateReferral with duplicate referred_id returns unique constraint error
- Test FindUserIDByReferralCode returns correct user ID
- Test FindUserIDByReferralCode with unknown code returns ErrNotFound

These are integration tests gated by `os.Getenv("DATABASE_URL")`.

### Task 2: Add `ref` field to registration handler

**Files:**
- `backend/internal/api/handlers/auth.go` — modify `RegisterRequest` and `Register` method
- `backend/internal/api/handlers/auth_test.go` — add test cases

Changes to `RegisterRequest`:
```go
type RegisterRequest struct {
    Email       string `json:"email"`
    Password    string `json:"password"`
    Username    string `json:"username"`
    DisplayName string `json:"display_name"`
    Ref         string `json:"ref,omitempty"` // optional referral code
}
```

Changes to `AuthHandlers`:
- Add `referralRepo ReferralRepositoryForAuth` field to struct
- Define interface:
  ```go
  type ReferralRepositoryForAuth interface {
      FindUserIDByReferralCode(ctx context.Context, code string) (string, error)
      CreateReferral(ctx context.Context, referrerID, referredID string) error
  }
  ```
- Update `NewAuthHandlers` to accept the referral repo (add parameter)
- In `Register()`, AFTER successful user creation + auth_method creation (after Step 6.5), add:
  ```
  Step 6.6: If req.Ref is non-empty:
    1. Look up referrer by code (FindUserIDByReferralCode)
    2. If found AND referrerID != createdUser.ID (no self-referral):
       - Call CreateReferral(ctx, referrerID, createdUser.ID)
       - If error, log warning and continue (do NOT fail registration)
    3. If not found or error, log and continue silently
  ```

**Tests (TDD):** `backend/internal/api/handlers/auth_test.go`
- Test registration with valid `ref` field -> 201 + referral created (verify via mock)
- Test registration with unknown `ref` -> 201, no referral created, no error
- Test registration with empty `ref` -> 201, referral logic not invoked
- Test registration with self-referral `ref` (referrer = new user) -> 201, no referral created
- Test referral creation failure does not block registration -> 201

Mock: add `mockReferralRepoForAuth` implementing `ReferralRepositoryForAuth`.

### Task 4: Frontend — read `?ref=CODE` and forward to API

**Files:**
- `frontend/app/join/page.tsx` — read query param, pass to register call
- `frontend/lib/api.ts` — add `ref` parameter to `register()` method
- `frontend/hooks/use-auth.tsx` — add `ref` parameter to `register` callback

Changes to `frontend/lib/api.ts` `register()`:
```typescript
async register(email: string, password: string, username: string, displayName: string, ref?: string): Promise<...> {
    return this.fetch<...>('/v1/auth/register', {
      method: 'POST',
      body: JSON.stringify({ email, password, username, display_name: displayName, ...(ref && { ref }) }),
    });
}
```

Changes to `frontend/hooks/use-auth.tsx`:
- Add `ref?: string` parameter to the `register` function in `AuthContextType` and the `useCallback`
- Pass it through to `api.register(..., ref)`

Changes to `frontend/app/join/page.tsx`:
- Import `useSearchParams` from `next/navigation`
- Read `const ref = searchParams.get('ref')` at component top
- Pass `ref` to `register(email, password, username, displayName, ref || undefined)`
- Also pass `ref` as a query param when redirecting through OAuth flows (append `?ref=CODE` to OAuth redirect state/return URL) — **SCOPE NOTE**: OAuth referral tracking is a stretch goal. If it requires significant changes to the OAuth flow (state parameter threading), defer to a future phase. The primary path is email registration.
- Wrap the page body in `<Suspense>` if needed for `useSearchParams()` (Next.js 15 requirement)

**Tests (TDD):** `frontend/app/join/__tests__/page.test.tsx` (or similar)
- Test: when URL has `?ref=ABC123`, the register API call includes `ref: "ABC123"` in the body
- Test: when URL has no `ref` param, the register API call does not include `ref`

Use Vitest with `vi.mock()` for the API client. Mock `useSearchParams` to control the query param.

## Acceptance Criteria (from ROADMAP)

1. Registration API accepts optional `ref` field in request body -> **Task 2**
2. Valid `ref` on registration -> exactly one row in `referrals` linking referrer and new user -> **Tasks 1 + 2**
3. Invalid/unknown `ref` -> silently ignored, registration returns 201 -> **Task 2**
4. `/join?ref=CODE` frontend extracts and forwards `ref` to API -> **Task 4**
5. Integration test: register with valid `ref`, query `referrals`, exactly one row -> **Task 1 (integration tests)**

Task 1 (ReferralRepository methods + tests)

## Scope Boundaries

**In scope:**
- Email/password registration referral tracking
- ReferralRepository.CreateReferral + FindUserIDByReferralCode
- Frontend /join page reading ?ref query param

**Out of scope (defer to future):**
- OAuth (GitHub/Google) referral tracking — would require threading ref through OAuth state parameter
- Referral dashboard UI (Phase 9)
- Referral rewards/badges (v2)
